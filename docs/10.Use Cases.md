#### 1 Сценарий "Установка API"

UC-01: Установка API-ключа  
Актор: Пользователь  
Цель: Сохранить валидный API-ключ для работы системы  
Precondition: Программа запущена  

Основной поток:
1. Пользователь открывает вкладку "Настройки".
2. Система открывает форму ввода API-ключа.
3. Пользователь вводит API-ключ.
4. Система отправляет тестовый запрос в LLM API.
5. Система получает успешный ответ от провайдера.
6. Система шифрует и сохраняет API-ключ.

Альтернативные потоки:
- 4А. LLM API возвращает `401` → система показывает ошибку: "Неверный API-ключ" и возвращает пользователя к форме ввода.
- 4Б. LLM API возвращает `402` или `429` → система показывает ошибку: "Недостаточно средств или превышен лимит" и возвращает пользователя к форме ввода.
- 4В. Провайдер недоступен или не отвечает → система показывает ошибку: "Провайдер недоступен" и возвращает пользователя к форме ввода.

Postcondition: API-ключ сохранён, система готова отправлять запросы к LLM.

#### 2 Сценарий "Настройка итерации"

UC-02: Настройка итерации  
Актор: Пользователь  
Цель: Задать количество итераций дискуссии  
Precondition: Программа запущена, чат создан и открыт  

Основной поток:
1. Пользователь открывает настройки итерации.
2. Система показывает поле ввода количества итераций.
3. Пользователь вводит значение.
4. Система валидирует и сохраняет значение.

Альтернативные потоки:
- 3А. Пользователь вводит неположительное число → система показывает ошибку: "Введите число больше 0" и оставляет пользователя в настройках итерации.

Postcondition: Количество итераций сохранено.

#### 3 Сценарий "Запуск дискуссии"

UC-03: Запуск дискуссии  
Актор: Пользователь  
Цель: Запустить итерацию и получить ответы агентов дискуссии  
Precondition: Программа запущена, API-ключ сохранён, чат создан и открыт, активен хотя бы один агент дискуссии  

Основной поток:
1. Пользователь вводит запрос в поле ввода.
2. Пользователь нажимает кнопку "Отправить".
3. Система создаёт новую итерацию.
4. Система последовательно отправляет запросы активным агентам через LLM API.
5. Система отображает потоковую выдачу ответа каждого агента.
6. Система сохраняет ответы агентов в JSON-лог сессии.
7. После завершения итерации система предлагает пользователю выбрать дальнейшее действие.

Альтернативные потоки:
- 4А. Провайдер не отвечает → система показывает ошибку: "Провайдер недоступен".
- 4Б. LLM API возвращает `402` или `429` → система показывает ошибку: "Недостаточно средств или превышен лимит".
- 4В. LLM API возвращает `400` из-за переполнения контекста → система показывает ошибку: "Контекст переполнен, очистите историю".

Postcondition: Ответы агентов сохранены, текущая итерация завершена и ожидает дальнейшего действия пользователя.

#### 4 Сценарий "Остановка дискуссии"

UC-04: Остановка дискуссии  
Актор: Пользователь  
Цель: Остановить текущую дискуссию  
Precondition: Дискуссия запущена, итерация находится в статусе `In_Process`  

Основной поток:
1. Пользователь нажимает кнопку остановки.
2. Система показывает окно с тремя вариантами действия.
3. Пользователь выбирает "Продолжить".
4. Система продолжает выполнение текущей итерации.

Альтернативные потоки:
- 3А. Пользователь выбирает "Завершить + Синтезатор" → система завершает итерацию и запускает Синтезатор.
- 3Б. Пользователь выбирает "Завершить и остановить" → система завершает итерацию, переводит сессию в паузу и не запускает Синтезатор.

Postcondition:
- основной поток: дискуссия продолжается;
- 3А: итерация получает статус `Synthesized`, итоговый ответ сформирован;
- 3Б: итерация получает статус `Stopped`, сессия поставлена на паузу.
